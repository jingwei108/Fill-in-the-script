# 量化评教自动填写脚本设计计划

## 1. 需求分析

### 1.1 目标
为学校教务系统（`https://jw.hstc.edu.cn/eams/`）开发一个自动化脚本，实现量化评教快速填写。

### 1.2 填写规则
- 全部评分题目中，**固定 1 题选择“良”**
- **其余题目全部选择“优”**
- 末尾意见建议题填写 **“无”**
- 自动提交表单，并继续填写下一门未评课程

### 1.3 约束条件
- 教务系统需要账号登录（脚本不处理登录，需用户手动登录）
- 脚本应在用户已登录后的页面运行
- 不能修改学校服务器数据逻辑，仅模拟用户点击

---

## 2. 技术方案

### 2.1 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Tampermonkey 用户脚本** | 无需安装环境、即装即用、与浏览器登录态天然共享 | 依赖页面 DOM 结构，若改版需更新 | 普通用户快速使用 |
| **Python + Playwright** | 稳定、可后台运行、可批量处理、可控性强 | 需要安装 Python 和浏览器驱动 | 技术用户、批量自动化 |

### 2.2 推荐方案
优先使用 **Tampermonkey 浏览器脚本**，其次提供 **Python + Playwright** 方案作为备选。

---

## 3. 实现逻辑

### 3.1 页面识别
脚本需要识别两种页面：
1. **列表页**：包含所有课程的“进行评教”链接
2. **详情页**：包含单选评分题和建议文本框

识别方式：
- 列表页：页面中存在文本包含“进行评教”的 `<a>` 元素
- 详情页：页面中存在大量 `input[type="radio"]`（>5 个）

### 3.2 自动填写流程

```
开始
│
├─ 判断是否已登录 → 未登录则等待/提示
│
├─ 识别当前页面类型
│   ├─ 列表页：添加“一键自动评教”按钮
│   └─ 详情页：执行填写逻辑
│
├─ 详情页填写逻辑
│   ├─ 获取所有 radio 分组（按 name 分组）
│   ├─ 根据配置确定哪一题填“良”
│   ├─ 遍历每个分组：指定题选“良”，其余选“优”
│   ├─ 填写建议文本框为“无”
│   └─ 点击提交按钮
│
└─ 提交后返回列表页，继续下一门
```

### 3.3 题目定位策略

由于不同页面 DOM 结构可能不同，脚本采用多层兜底策略：

1. **首选**：通过 `<label>` 文字匹配“优/良/中/合格/不合格”并点击
2. **兜底**：查找 radio 父元素的文字，匹配后点击 radio
3. **最终兜底**：直接遍历同 name 的 radio，按 value 或相邻文本匹配

### 3.4 建议框定位策略

1. 按 `name` 关键词匹配（suggest/advice/opinion/remark/comment）
2. 按元素类型匹配所有 `textarea`
3. 按 placeholder 关键词匹配（意见、建议、其他）

### 3.5 提交按钮定位策略

1. 按选择器匹配：`input[type="submit"]`、`button[type="submit"]` 等
2. 按按钮文字匹配：包含“提交”“保存”“确定”

## 4. 配置项设计

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `suggestionText` / `SUGGESTION_TEXT` | `"无"` | 建议内容 |
| `liangIndex` / `LIANG_INDEX` | `0` | 哪一题填“良”。`0`=第一题，`-1`=最后一题，`None`=随机 |
| `autoSubmit` / `AUTO_SUBMIT` | `True` | 是否自动提交 |
| `autoNext` / `AUTO_NEXT` | `True` | 是否自动继续下一门 |
| `delay` / `DELAY` | `0.8s` | 提交前等待时间，防止请求过快 |

## 6. 风险控制与注意事项

### 6.1 页面结构变更
学校教务系统页面若改版，脚本中的选择器可能失效。已在脚本中采用多层兜底策略，但仍需根据实际 DOM 微调。

### 6.2 账号安全
脚本不存储、不上传账号密码。Tampermonkey 脚本运行在浏览器本地，Playwright 脚本由用户本地执行。

### 6.3 使用建议
- 先在一门课程上手动测试脚本
- 确认提交结果正确后再批量运行
- 不建议在评教高峰期高频提交，避免触发反爬虫
